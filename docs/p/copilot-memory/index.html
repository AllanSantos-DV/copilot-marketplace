copilot-memory

v1.9.0

O copilot-memory dá ao agente uma memória que dura além da sessão, escopada por projeto. Ele injeta o contexto certo no começo de cada conversa (decisões, convenções, pendências), deixa você buscar e salvar conhecimento, e destila o que deu certo em skills reusáveis. É um cliente puro do servidor de memória native-java: descobre o daemon, reusa se estiver no ar e, se não estiver, baixa e sobe o servidor — verificando a assinatura antes de rodar. Toda a memória fica na sua máquina. Ele também impõe os passos que dá pra esquecer: desde a v1.8, um gate barra o git push até um revisor externo independente aprovar exatamente aquele commit. Novo na v1.9.0: um grafo semântico do código deixa o agente navegar um repositório — o aberto ou um externo gigante — em vez de garimpar arquivo por arquivo com grep.

⌁ copilot plugin install copilot-memory@copilot-marketplace
código↗

  • Navega o código, não garimpa

    Novo na v1.9.0. Sete tools graph_* montam um grafo semântico do repo — símbolos viram nós; chamadas, imports e contêm viram arestas; ranqueado por PageRank. O agente vai direto aos pontos-chave, sem varrer arquivo por arquivo. Vale pro projeto aberto ou pra um repo externo gigante.

  • Barra o git push sem revisão

    Desde a v1.8.0. Um gate opt-in por projeto torna passos críticos obrigatórios e à prova de esquecimento. O primeiro exige que um revisor externo independente aprove exatamente aquele commit antes de publicar — com recibo amarrado ao SHA. Determinístico onde importa, semântico onde agrega.

  • Lembra por projeto, não vaza entre eles

    Cada leitura e escrita é carimbada com o project_id do projeto aberto. A memória de um projeto nunca aparece em outro.

  • Recall em dois níveis

    Skills voltam como ponteiro (nome + descrição + id) e o agente abre o corpo só quando precisa; fatos voltam inline, prontos para usar. Menos ruído, mais contexto.

  • Aprende com a sessão, sozinho

    Um agente curador lê a conversa (checkpoints + turnos) em background e destila lições técnicas e comportamentais — semanticamente, entendendo ironia e crítica, não por palavra-chave. Skills de projeto entram no recall automaticamente.

  • Sobe o servidor sozinho

    Sem daemon e sem servidor instalado, o plugin baixa a release pública, confere o sha256 e sobe o servidor. Se já existe, apenas reusa. Requer Java 21.

  • Um painel pra ver tudo

    Um canvas lateral mostra, ao vivo, a saúde do daemon, o escopo do projeto (com a escada que resolveu o project_id), os documentos e skills recentes e a telemetria de recall.

  • Muda o escopo sem perder a memória

    Ao declarar um project_id novo, o plugin detecta a memória do escopo antigo e migra com a sua aprovação — reatribui sem re-processar, e avisa quando há risco de deixar memória órfã.

#O que é

O copilot-memory é uma extensão do GitHub Copilot que liga o agente ao seu servidor de memória local (o native-java). No começo da sessão e a cada pergunta, ela busca na memória o que é relevante para o projeto aberto e injeta isso no contexto — então o agente já chega sabendo as decisões de arquitetura, as convenções e as pendências que ficaram de antes.

O plugin não é o servidor: é um cliente. Toda a inteligência — como a memória é composta, escopada e reforçada — vive no servidor. O plugin descobre o daemon, consome o recall e escreve de volta, sempre carimbando o escopo do projeto.

i

A memória é local: fica na sua máquina, no servidor native-java. O plugin só conversa com o daemon local por HTTP.

#Como usar

  1. Instale o plugin

    Pelo marketplace, com o comando da seção Instalar abaixo.

  2. Deixe o servidor subir

    Se não houver um servidor de memória rodando, o plugin baixa e sobe na primeira sessão. Requer Java 21 no PATH. Ou rode memory_setup para provisionar na hora.

  3. Trabalhe normalmente

    O contexto do projeto entra sozinho no início da sessão e a cada pergunta. Use memory_search para buscar e memory_save para guardar uma decisão ou nota.

  4. Dê um escopo estável ao projeto

    Se o projeto não tem .memory/project.json nem remote git, o plugin sugere criar um — uma vez. Aceite (ou use memory_init_project) para a memória casar entre máquinas.

  5. Destile o que aprendeu

    Depois de um marco confirmado, chame memory_distill: o agente extrai o que é reusável e verificado como skill candidata, que você promove com memory_promote_skill.

  6. Blinde o que não pode esquecer (opcional)

    Nos repos que quiser, ligue um gate com memory_gate enable: a partir daí, o git push sem revisão externa fica bloqueado. Rode memory_gate_review para liberar o commit. É opt-in — vale só naquele repositório.

#Gates obrigatórios

Revisar o código antes de publicar é fácil de esquecer — e "lembrar" não é garantia de nada. Os gates tiram os passos críticos da disciplina humana e os tornam obrigatórios: o plugin bloqueia a ação até a condição ser cumprida. O primeiro gate pronto é revisor externo antes do git push — tenta publicar sem revisão? O push para na hora, com uma mensagem que diz o que fazer:

$ git push origin main

  ⛔  Gate obrigatório — revisão externa antes do push
      commit d4f9c1a ainda não foi revisado; nada foi enviado.

      → rode  memory_gate_review  para revisar o diff do push
        e liberar exatamente este commit.

  1. Ligue por repositório

    memory_gate enable escreve um hook no .github/hooks/hooks.json daquele repo. É opt-in: suas outras sessões, projetos e máquinas não mudam em nada.

  2. O hook barra o push

    Um hook determinístico intercepta a chamada de git push do agente. Sem um recibo válido para aquele commit, ele nega e nada é enviado. Nada de regex frágil: ele reconhece a forma da ação (que aquilo é um push), não o significado.

  3. Rode a revisão externa

    memory_gate_review dispara um revisor externo independente — outro modelo, num subprocesso separado, sem o contexto do agente barrado (anti "echo chamber"). Ele lê o diff do que seria publicado e caça bugs reais, falhas de segurança e segredos vazados.

  4. Limpo? Ganha um recibo preso ao commit

    Se o parecer sai limpo, a tool grava um recibo amarrado ao SHA do commit. Só então o git push daquele commit passa. Quem publica não assina o próprio recibo — quem assina é a revisão independente.

  5. Commitou de novo? Revisa de novo

    Emendou ou fez um commit novo? O SHA muda, o recibo perde a validade e o gate pede nova revisão. Não dá para passar com um parecer velho.

i

Provado ao vivo. O gate já barrou um git push de verdade numa sessão real, e o revisor externo pegou problemas concretos — uma credencial exposta e uma lógica quebrada — e recusou o recibo. Não é teoria.

!

Emergência? Existe um break-glass auditado: um override que libera o push na hora e registra quem liberou, por quê e quando expira. Ele só entra com a sua ordem explícita — o agente nunca o aciona sozinho.

Por que isto é sólido: o bloqueio é determinístico onde precisa (um hook, não um palpite) e a análise é semântica onde agrega (a revisão por IA) — nada de regex frágil decidindo o que barrar. As regras vivem como dado (políticas + recibos locais), não num hook cravado que vira manutenção infinita: o hook é um despachante fino e estável, então novos gates entram sem reescrever nada. E o bloqueio roda num hot-path local — lê políticas e recibos do disco, sem depender do daemon.

Isso é só o começo. Os próximos gates já estão no mapa — aprovação antes de um release, um agente de design antes de publicar onde há GitHub Pages — e o próprio autoaprendizado pode propor um gate novo a partir do que você corrige, para você promover.

#O escopo do projeto (.memory/project.json)

A memória só serve se o escopo for estável. O plugin resolve o project_id nesta ordem: o .memory/project.json do projeto (a intenção declarada), depois o remote git, depois o repositório base (que mantém o escopo estável entre worktrees), e só então o caminho da pasta.

Quando o escopo é frágil — uma pasta sem git nem .memory/project.json — o project_id vira o caminho da pasta, que muda de máquina para máquina e faz o recall não casar. Nesse caso o plugin sugere, uma única vez por projeto, que o agente analise a estrutura e crie o .memory/project.json com um id canônico (por exemplo owner/projeto).

i

A sugestão nunca insiste: se você recusar, ou se o .memory/project.json já existir, ela não aparece de novo. A decisão fica guardada por projeto, fora da pasta.

#Recall em dois níveis

Nem todo conhecimento entra no contexto do mesmo jeito. O plugin injeta em dois níveis, para dar contexto sem inchar o prompt:

  • Skills = ponteiro. Uma skill volta como nome + descrição + id. O agente decide se abre o corpo completo com memory_get — só carrega o que vai usar.
  • Fatos = inline. Uma decisão ou nota volta com o texto direto no contexto, pronto para o agente usar sem um passo a mais.

Se a composição do recall demora ou falha, o plugin cai para uma busca escopada mais barata — sempre pelo project_id, nunca uma busca aberta que vazaria de outros projetos. E um teto de tempo garante que o recall nunca trave a sua sessão.

#O destilador de aprendizado (curadoria por agente)

O destilador fecha o ciclo: a experiência da sessão vira conhecimento reusável. Quem faz isso é um agente curador — um modelo que lê a conversa e entende o que aconteceu, não um filtro de palavras-chave. Roda sozinho, em background, quando a sessão abre.

  • Lê a conversa, não a máquina. A fonte são os checkpoints do Copilot (que já são um resumo curado da sessão) e os turnos vivos, limpos para conter só o que você disse e o que o assistente respondeu — sem ruído de ferramentas e hooks.
  • Semântico, não regex. O curador entende ironia, uma reclamação, um elogio, uma instrução repetida. É assim que ele sabe se algo deu certo ou errado — coisas que uma busca por palavra mágica nunca pegaria.
  • Dois tipos de lição. Técnicas (uma API que se comporta diferente, um passo que resolveu o problema) E comportamentais: anti-padrões do próprio assistente que você criticou, para ele não repetir.
  • Não duplica, reconcilia. Cada lição vira uma busca semântica na memória (do projeto e global) e uma decisão: criar nova, atualizar uma existente (corrigindo contradições em vez de duplicar), evoluir para global (quando serve a qualquer projeto) ou ignorar se já existe.
  • Incremental e sem retrabalho. Cada checkpoint e bloco de turnos é marcado por id ao ser curado, então o background nunca refaz o que já processou.
  • Errou? Sai do recall. memory_invalidate_skill tira uma lição errada do recall preservando o histórico (auditável, reversível).
i

Skills de projeto entram no recall automaticamente — o curador é o filtro de qualidade. Sua revisão fica reservada ao que sobe para o escopo global (que afeta todos os projetos). Segredos são redigidos antes de qualquer curadoria.

#Sobe o servidor sozinho

Um plugin de memória sem servidor não serve para nada. Por isso, quando não há daemon vivo e o servidor não está instalado, o copilot-memory faz o primeiro arranque por você:

  • Baixa a release pública do mcp-memory-server (o repositório oficial de releases) por HTTPS.
  • Confere o sha256 do arquivo antes de usar — um download corrompido ou adulterado é recusado e nada sobe.
  • Sobe o servidor destacado, que passa a se auto-anunciar e a cuidar de si (instância única, atualizações). A partir daí o plugin só descobre e reusa.
  • Degrada em silêncio se faltar Java ou rede: o plugin nunca trava a sessão — no pior caso, fica sem memória e avisa.
!

Instalar o plugin autoriza esse download na ausência de um servidor — é parte esperada do setup. Para desligar, defina COPILOT_MEMORY_AUTOPROVISION=0: o plugin então só reusa um daemon já existente.

#O painel (canvas)

Chame memory_dashboard (ou abra "Memory" na seção de canvas do app) para ver a memória do projeto num painel lateral, ao vivo e com auto-refresh. Ele não muda nada — é só leitura.

  • Saúde do daemon — online, offline ou degradado, com a URL e a versão do servidor.
  • Escopo — o project_id e a *escada* que o resolveu (declarado → git remote → repo base → caminho → nome), destacando qual venceu e avisando quando o escopo é frágil.
  • Documentos e skills — os itens recentes e as skills (ativas e candidatas) do projeto, além de uma busca escopada.
  • Telemetria de recall — quantos recalls, quantos ponteiros injetados, quantas buscas de corpo e o *hit-rate* ponteiro→fetch.
i

Com o daemon offline, o painel oferece um botão para provisionar o servidor — o mesmo caminho consentido do memory_setup. O painel aparece depois de reiniciar o app uma vez (o hook espelha o plugin para as extensões do app).

#Como funciona

  • Descoberta — lê o registro do daemon (~/.mcp-memory/run/daemon.json), faz um health-check e reusa a URL. Offline, degrada com aviso.
  • Escopo — resolve o project_id pelo .memory/project.json ou pelo git, e carimba toda leitura/escrita com ele.
  • Recall — nos hooks de início de sessão e de cada pergunta, compõe o contexto (ponteiro para skills, inline para fatos) e injeta.
  • Escrita e skills — memory_save guarda conhecimento; o ciclo save_skill → promote → invalidate cura as skills reusáveis.
  • Gates (opt-in) — nos repos que você habilita, um hook determinístico intercepta o git push e o barra até memory_gate_review gravar o recibo do commit. O bloqueio lê políticas e recibos locais: hot-path, sem daemon.

O plugin não recria o que o servidor já faz — composição do recall, telemetria de uso, reforço e poda são do servidor. Ele é o cliente que traz esse valor para dentro da sua sessão do Copilot.

#Perguntas frequentes

Preciso instalar o servidor de memória na mão?

Não. Se não houver um servidor rodando nem instalado, o plugin baixa a release pública, confere o sha256 e sobe o servidor na primeira sessão — ou na hora, se você rodar memory_setup. Precisa de Java 21 no PATH.

A memória de um projeto aparece em outro?

Não. Toda leitura e escrita é escopada pelo project_id. O plugin nunca faz uma busca aberta — mesmo o plano B de recall é escopado, então a memória de um projeto não vaza para outro.

Por que ele pede para criar o .memory/project.json?

Porque sem ele (e sem um remote git) o escopo vira o caminho da pasta, que muda entre máquinas e faz o recall não casar. Com o arquivo, o project_id é declarado e estável. A sugestão aparece no máximo uma vez por projeto e nunca insiste.

O que o destilador salva na memória?

Só o que é generalizável e verificado. Ele exige um sinal concreto (uma execução de ferramenta bem-sucedida, a sua confirmação) e redige segredos antes de propor. Sem prova, recusa criar a skill. E toda skill nasce candidata, fora do recall automático, até você promovê-la.

O plugin fica lento se o servidor demora?

Não trava a sessão. O recall tem um teto de tempo e cai para uma busca escopada mais barata se a composição demora; se estourar o teto, ele injeta nada e segue. Os hooks nunca derrubam a sua conversa.

Como desligo o download automático do servidor?

Defina COPILOT_MEMORY_AUTOPROVISION=0. O plugin então apenas reusa um daemon que já esteja rodando e degrada se não houver — sem baixar nem subir nada.

Onde mora o código do servidor?

No projeto native-java (mcp-memory-server), publicado como release pública. Esta vitrine guarda só o plugin cliente; o servidor é baixado do repositório oficial de releases, verificado por sha256.

Tem uma tela para ver o estado da memória?

Tem. O memory_dashboard abre um painel lateral (canvas) com a saúde do daemon, o escopo do projeto e a escada que resolveu o project_id, os documentos e skills recentes e a telemetria de recall. É só leitura e aparece depois de reiniciar o app uma vez.

Se eu declarar um project_id novo, perco a memória antiga?

Não. Ao declarar um .memory/project.json, o plugin detecta a memória carimbada com o escopo anterior e oferece migrá-la para o novo id, via reatribuição sem re-processar (o conteúdo é preservado). A migração nunca é automática: ele previsualiza e só aplica com a sua confirmação, e avisa quando o escopo antigo é compartilhado (git) para não deixar a memória órfã para colegas.

Os gates afetam meus outros projetos?

Não. O gate é opt-in por repositório: memory_gate enable escreve um hook só no .github/hooks/hooks.json daquele repo. Suas outras sessões, projetos e máquinas seguem intactos. memory_gate disable remove quando quiser.

Se o daemon da memória cair, o gate para de barrar?

Não. O bloqueio roda num hot-path local: o hook lê as políticas e os recibos direto do disco (em ~/.copilot-memory/), sem rede e sem daemon. Só a revisão (memory_gate_review) usa o modelo — o bloqueio em si é determinístico e independente.

Preciso do painel (canvas) aberto para o gate valer?

Não. O gate é um hook que dispara toda vez que o agente tenta um git push, com ou sem painel aberto. O canvas é só leitura e não tem relação com o bloqueio.
